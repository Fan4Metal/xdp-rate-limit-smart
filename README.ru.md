# xdp-rate-limit-smart

[English](README.md) · **Русский**

XDP-фильтр для Ubuntu, который считает входящий IPv4-трафик по source IP и общий
трафик интерфейса. Python-демон добавляет «умный режим»: когда общий входящий
трафик превышает заданный порог, он банит только те IP, которые сами дают больше
заданного индивидуального порога.

IPv6 в этой версии пропускается без фильтрации — см.
[Ограничения и модель угроз](#ограничения-и-модель-угроз).

## Что реализовано

- XDP-программа на C.
- Загрузка через маленький libbpf-loader, без BCC.
- Per-CPU счётчики (без межъядерной конкуренции за кэш-линию и без атомиков).
- Общая статистика по интерфейсу: Mbps и PPS.
- Статистика по каждому source IPv4.
- **Smart global mode** ([подробнее](#smart-mode)):
  - общий трафик выше `global_mbps_limit` / `global_pps_limit`;
  - банятся IP выше `smart_ban_min_mbps` / `smart_ban_min_pps`.
- Опциональный прямой per-IP лимит через `per_ip_mbps_limit` / `per_ip_pps_limit`
  ([подробнее](#прямой-per-ip-лимит)).
- Временный бан с авторазбаном.
- Whitelist IPv4 и CIDR, например `1.2.3.4` или `10.0.0.0/24`.
- systemd-сервис.

## Как это устроено

Быстрый путь и политика разделены:

- **XDP-программа** ([`src/xdp_rate_limit.bpf.c`](src/xdp_rate_limit.bpf.c)) только
  считает байты/пакеты и дропает source-адреса из чёрного списка. Её загружает и
  пинит map'ы **loader** ([`src/xdp_loader.c`](src/xdp_loader.c)).
- **Python-демон** ([`src/xdp_rate_daemon.py`](src/xdp_rate_daemon.py)) раз в
  `interval_seconds` читает запиненные map'ы, считает скорости и решает, кого банить.

Всё связывает systemd-шаблон
([`systemd/xdp-rate-limit@.service`](systemd/xdp-rate-limit@.service)), который
вызывает обёртку ([`src/xdp-rate-limit-wrapper`](src/xdp-rate-limit-wrapper)).

> Подробный разбор внутренностей по компонентам — путь пакета в XDP, BPF-карты,
> жизненный цикл загрузчика и цикл решений демона — см. в
> **[Архитектура и механизм работы](docs/architecture.ru.md)**.

## Требования

- Ядро Linux ≥ 4.10 (нужно для `LRU_PERCPU_HASH`; любая актуальная Ubuntu подходит).
- Права root и монтируемый bpffs в `/sys/fs/bpf`.
- Тулчейн для сборки (ставится автоматически в [`install.sh`](install.sh)):
  `clang`, `llvm`, `gcc`, `make`, `libbpf-dev`, `libelf-dev`, `zlib1g-dev`,
  `iproute2`, `python3`.

## Установка

> **Замечание:** во всех командах `eth0` — это заглушка для вашего интерфейса;
> подставляйте своё имя (например, `ens3`) везде, в том числе в имени
> systemd-юнита `xdp-rate-limit@<iface>`. Своё имя смотрите через `ip -br link`.

```bash
sudo ./install.sh eth0
```

Где `eth0` — внешний интерфейс сервера. Посмотреть свой:

```bash
ip -br link
ip route get 1.1.1.1
```

Установщик копирует дефолтный [`etc/config.json`](etc/config.json) в
`/etc/xdp-rate-limit/config.json` и, если распознаёт текущего SSH-клиента,
автоматически добавляет его IP в whitelist. Всё равно перепроверьте whitelist —
см. [Важные замечания](#важные-замечания).

## Обновление

Чтобы обновить уже установленную версию до новой, подтяните свежие исходники и
заново запустите установщик со своим интерфейсом. `install.sh` перекомпилирует
BPF-объект и загрузчик, переустанавливает демон, обёртку и systemd-юнит, а ваш
существующий `/etc/xdp-rate-limit/config.json` **не трогает**. Чтобы загрузились
новые бинарники, нужен перезапуск — повторный запуск установщика не перезапускает
уже работающий сервис:

```bash
cd xdp-rate-limit-smart
git pull
sudo ./install.sh eth0
sudo systemctl restart xdp-rate-limit@eth0
```

Затем убедитесь, что сервис поднялся чисто, и проверьте лог — `NRestarts` не должен
расти (см. [Проверка состояния и диагностика](#проверка-состояния-и-диагностика)):

```bash
systemctl status xdp-rate-limit@eth0
sudo journalctl -u xdp-rate-limit@eth0 -n 20 --no-pager
```

Перезапуск чисто отсоединяет и заново присоединяет XDP, так что его безопасно
выполнять на живом сервере. Он сбрасывает активные баны — запиненные карты
удаляются при остановке и создаются пустыми при старте, — но это безвредно: любой
источник, всё ещё превышающий лимит, будет перебанен за один-два тика (баны и так
временные, а `/sys/fs/bpf` не переживает перезагрузку). Если вы поменяли только
Python-демон и хотите пропустить перекомпиляцию, можно переустановить лишь этот
файл вместо полного установщика:

```bash
sudo install -m0755 src/xdp_rate_daemon.py /usr/local/sbin/xdp-rate-daemon
sudo systemctl restart xdp-rate-limit@eth0
```

## Настройка

```bash
sudo nano /etc/xdp-rate-limit/config.json
```

Агрессивный вариант — общий порог 5 Мбит/с, банить IP от 2 Мбит/с:

```json
{
  "interval_seconds": 1.0,

  "smart_global_enabled": true,
  "global_mbps_limit": 5.0,
  "global_pps_limit": 0,

  "smart_ban_min_mbps": 2.0,
  "smart_ban_min_pps": 0,

  "per_ip_mbps_limit": 0,
  "per_ip_pps_limit": 0,

  "ban_seconds": 300,
  "extend_ban_on_repeat": true,
  "max_bans_per_tick": 50,

  "whitelist": [
    "127.0.0.1",
    "ВАШ_SSH_IP"
  ],

  "dry_run": false
}
```

Более мягкий вариант — общий порог 10 Мбит/с, банить IP от 5 Мбит/с:

```json
{
  "smart_global_enabled": true,
  "global_mbps_limit": 10.0,
  "smart_ban_min_mbps": 5.0,
  "per_ip_mbps_limit": 0,
  "ban_seconds": 300,
  "whitelist": ["127.0.0.1", "ВАШ_SSH_IP"]
}
```

Перед применением проверьте, что JSON валиден — иначе демон оставит старый конфиг
и залогирует `tick failed`:

```bash
python3 -m json.tool /etc/xdp-rate-limit/config.json >/dev/null && echo "JSON OK"
```

Демон перечитывает файл **на лету** при изменении `mtime` — рестарт не нужен,
новые значения применяются в течение `interval_seconds` (в логе появится строка
`Config loaded: ...`). Whitelist тоже синхронизируется автоматически. Принудительно
перезапустить, если нужно:

```bash
sudo systemctl restart xdp-rate-limit@eth0
```

## Smart mode

### Идея

Обычный per-IP лимит наказывает любого «тяжёлого» клиента, даже когда сервер
простаивает и никому не мешает. Smart mode вместо этого задаёт по очереди два
вопроса и банит IP, **только если оба ответа «да»**:

1. **Реально ли интерфейс под нагрузкой?** — *глобальный шлюз*: общий входящий
   трафик должен превысить `global_mbps_limit` (или `global_pps_limit`).
2. **Является ли этот конкретный IP заметной частью нагрузки?** — *per-source
   шлюз*: скорость самого IP должна превысить `smart_ban_min_mbps` (или
   `smart_ban_min_pps`).

Пока канал спокоен, даже довольно активный клиент не трогается. И только когда
суммарный трафик становится проблемой, демон начинает банить крупнейших
«вкладчиков». В этом ключевое отличие от
[прямого per-IP лимита](#прямой-per-ip-лимит), который срабатывает безусловно.

### Как работает один тик

Каждые `interval_seconds` демон:

1. Читает глобальные и по-IP счётчики и пересчитывает их в Mbps/PPS за прошедший
   интервал.
2. Проверяет глобальный шлюз. Если он **не** превышен (или `smart_global_enabled`
   = `false`), smart-банов в этот тик нет.
3. Если превышен — каждый источник выше per-source шлюза становится кандидатом на
   бан (`reason=smart-global`).
4. Кандидаты сортируются по скорости (сначала самые крупные) и банятся, не более
   `max_bans_per_tick` за тик, каждый на `ban_seconds`.
5. Баны истекают автоматически; демон снимает их и логирует `UNBAN`.

Порог со значением `0` — **выключен**: например `global_pps_limit: 0` означает,
что для глобального шлюза используется только Mbps. Лимит срабатывает, когда
измеренное значение `>=` ему.

### Параметры

| Ключ | Значение |
| --- | --- |
| `smart_global_enabled` | Общий выключатель smart-режима. |
| `global_mbps_limit` / `global_pps_limit` | Глобальный шлюз: суммарные значения интерфейса, которые надо превысить до любого smart-бана. `0` отключает метрику. |
| `smart_ban_min_mbps` / `smart_ban_min_pps` | Per-source шлюз: банятся только IP выше него, когда глобальный шлюз открыт. `0` отключает метрику. |
| `ban_seconds` | Длительность бана; `0` = постоянный, пока не снят. |
| `extend_ban_on_repeat` | Если `true`, всё ещё нарушающему IP бан продлевается, а не истекает. |
| `max_bans_per_tick` | Предохранитель: сколько IP можно забанить за один интервал. |
| `interval_seconds` | Интервал измерения/принятия решения. |

### Как подобрать значения

1. Запустите с `"dry_run": true` (см. [Dry-run](#dry-run)) на какое-то время и
   понаблюдайте за периодической сводкой в [логах](#логи) — так вы узнаете свой
   обычный `global=...Mbps/...pps` и типичные скорости топ-IP.
2. Поставьте `global_mbps_limit` немного **выше** обычного пика — нужно, чтобы
   smart-режим включался при аномалии, а не в штатные загруженные периоды.
3. Поставьте `smart_ban_min_mbps` **выше** того, что обычно даёт один легитимный
   клиент, чтобы реальные пользователи не становились кандидатами даже при открытом
   глобальном шлюзе.
4. Для флудов пакетами / мелкими пакетами используйте PPS-шлюзы
   (`global_pps_limit` / `smart_ban_min_pps`), для флудов по полосе — Mbps-шлюзы.
   Можно задать оба: достаточно превышения любой метрики.
5. Держите свой admin/SSH IP в `whitelist` — адреса из него никогда не становятся
   кандидатами (см. [Важные замечания](#важные-замечания)).

### Разбор примеров

Глобальный шлюз **открыт** (итог 8 ≥ 5 Мбит/с), per-source шлюз на 2 Мбит/с:

```text
Общий входящий трафик eth0 = 8 Мбит/с   (global_mbps_limit = 5  -> шлюз ОТКРЫТ)
smart_ban_min_mbps = 2

1.1.1.1 = 3.2 Мбит/с  -> бан    (выше per-source шлюза)
2.2.2.2 = 2.4 Мбит/с  -> бан    (выше per-source шлюза)
3.3.3.3 = 0.4 Мбит/с  -> цел    (ниже per-source шлюза)
```

Глобальный шлюз **закрыт** — не банится никто, даже активный клиент:

```text
Общий входящий трафик eth0 = 3 Мбит/с   (global_mbps_limit = 5  -> шлюз ЗАКРЫТ)
smart_ban_min_mbps = 2

1.1.1.1 = 2.6 Мбит/с  -> цел    (сервер не под нагрузкой)
```

Если вам нужно забанить этого клиента на 2.6 Мбит/с независимо от общего уровня —
это ровно задача [прямого per-IP лимита](#прямой-per-ip-лимит). Правила работают
вместе: прямые лимиты проверяются всегда, smart-баны — только при открытом
глобальном шлюзе.

> **Саморегуляция by design.** Забаненный трафик дропается *до* учёта, поэтому
> забаненный IP перестаёт считаться в общем итоге. После бана худших нарушителей
> суммарный трафик интерфейса часто падает ниже `global_mbps_limit`, шлюз
> закрывается, и smart-режим прекращает банить дальше — см.
> [Учёт забаненного трафика](#учёт-забаненного-трафика).

## Прямой per-IP лимит

Если нужно банить IP всегда, даже без превышения общего лимита:

```json
{
  "per_ip_mbps_limit": 5.0,
  "per_ip_pps_limit": 0
}
```

Тогда любой IP выше 5 Мбит/с будет забанен сразу.

## Проверка состояния и диагностика

Состояние сервиса. Следите, чтобы `NRestarts` **не рос** — рост означает петлю
перезапусков (сервис падает и systemd поднимает его снова):

```bash
systemctl status xdp-rate-limit@eth0
systemctl show -p NRestarts -p ActiveState xdp-rate-limit@eth0
```

Что реально прикреплено к интерфейсу (должна висеть одна XDP-программа `xdp`):

```bash
sudo bpftool net show dev eth0
ip -d link show eth0 | grep -i xdp
```

Живой лог с банами и периодической сводкой:

```bash
sudo journalctl -u xdp-rate-limit@eth0 -f
```

Активные баны и размер скана демона (число source-IP в статистике):

```bash
# текущий чёрный список
sudo bpftool map dump pinned /sys/fs/bpf/xdp-rate-limit/eth0/blacklist_map
# сколько записей в статистике по source IP
sudo bpftool map dump pinned /sys/fs/bpf/xdp-rate-limit/eth0/stats_map | grep -c '"key"'
```

Сколько CPU ест сам демон прямо сейчас:

```bash
PID=$(systemctl show -p MainPID --value xdp-rate-limit@eth0)
top -b -n2 -d1 -p "$PID" | tail -5
```

Если сервис застрял в петле перезапусков или на интерфейсе осталась осиротевшая
XDP-программа — сбросьте счётчик и перезапустите. `load` самовосстанавливающийся:
при старте он снимает свой оставшийся экземпляр и цепляется заново, так что обычно
достаточно:

```bash
sudo systemctl reset-failed xdp-rate-limit@eth0
sudo systemctl restart xdp-rate-limit@eth0
```

## Dry-run

Для проверки без реального бана:

```json
{
  "dry_run": true
}
```

В логах будет `DRY-RUN BAN`, но IP не попадут в XDP blacklist.

## Остановка и удаление с интерфейса

Остановить:

```bash
sudo systemctl stop xdp-rate-limit@eth0
```

Отключить автозапуск:

```bash
sudo systemctl disable xdp-rate-limit@eth0
```

Принудительно снять XDP, если нужно:

```bash
sudo /usr/local/sbin/xdp-rate-loader unload eth0 /sys/fs/bpf/xdp-rate-limit/eth0
```

`unload` снимает только **свою** программу (по имени `xdp_rate_limiter`), в том
числе осиротевшую после потери пинов. Чужую XDP-программу он не трогает — чтобы
снять её принудительно, добавьте `--force` (или `XDP_FORCE=1` в окружении):

```bash
sudo /usr/local/sbin/xdp-rate-loader unload eth0 /sys/fs/bpf/xdp-rate-limit/eth0 --force
```

## Сборка из исходников

Чтобы собрать и проверить локально, не выкатывая на сервер, используйте
[`Makefile`](Makefile) (результат — в `./build`):

```bash
make            # компиляция BPF-объекта + loader
make pycheck    # байт-компиляция Python-демона
make verify     # дамп собранной BPF-программы (нужен bpftool)
make clean
```

BPF-объект и loader собираются только на Linux; `make pycheck` работает везде,
где есть `python3`.

## Важные замечания

1. Добавьте свой SSH/admin IP в `whitelist`, иначе можно забанить себя.
2. XDP работает на входящем трафике интерфейса.
3. Mbps считается по размеру пакета на входе в XDP — близко к интерфейсной
   скорости, но не идеально равно показаниям провайдера.
4. Whitelist не учитывается в статистике и не банится — см.
   [Учёт забаненного трафика](#учёт-забаненного-трафика).
5. При `ban_seconds = 0` бан будет постоянным, пока запись есть в map или сервис
   не перезапущен/не выгружен.

## Ограничения и модель угроз

Инструмент рассчитан на «шумных соседей» и умеренные всплески от небольшого числа
реальных источников. Чего он **не** делает:

1. **Спуфинг source IP / volumetric-флуд.** `stats_map` — это per-CPU LRU-хеш.
   При флуде с рандомизированных подделанных адресов записи вытесняются по LRU и
   пересоздаются с нуля, поэтому ни один такой IP не накапливает дельту до порога
   бана. Это принципиальное ограничение подхода «считать по source IP»: против
   классического спуф-флуда он слеп. Нужен апстрим-скрабер / BGP-blackhole или
   stateful-защита.
2. **IPv6 пропускается целиком** — без учёта и без бана. Атакующий по IPv6 обходит
   фильтр. Если IPv6 не нужен — закройте его файрволом отдельно.
3. **Реакция не мгновенная.** Бан выставляется по дельте за интервал (`~1 c`), плюс
   новый IP ловится только со второго тика. Микробёрсты короче интервала проходят.
4. **Ёмкость чёрного списка** — `blacklist_map` на 262144 записи. При переполнении
   демон логирует ошибку и перестаёт добавлять новые баны (существующие работают).
5. **Mbps считается на входе в XDP** — близко к, но не идентично показаниям
   провайдера.

## Учёт забаненного трафика

Активные IPv4 из чёрного списка проверяются до обновления `global_stats_map` и
`stats_map`. Поэтому пакеты, которые XDP уже дропает, не учитываются в
`global_mbps_limit`, не попадают в топ-кандидаты для smart-банов и сами по себе не
удерживают smart-режим включённым.
